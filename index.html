Material Complementar (IFC)

O objetivo deste material segregado é oferecer exemplos de requisições e algumas regras que facilitam o conteúdo do manual de forma complementar.

Exemplos de requisições e outras informações


01 - Pré requisitos / Introdução

O objetivo deste material, é fornecer as instruções necessárias para que os distribuidores estejam aptos para plugar em nossa plataforma e comercializar em nossos marketplaces.

A seguir tem por objetivo descrever todos os Web Services expostos pelo Middleware na plataforma Infrashop e os Web Services que devem ser expostos pelo Distribuidor para que seja possível realizar a integração do Distribuidor com o e-commerce.

De forma geral, a integração consiste em trocas de informações/dados entre a plataforma e sellers e vice e versa. Segue abaixo os fluxos de integrações:


Web services que devem ser disponibilizados pelo distribuidor:


Cliente (Protocolo Soap) - Do lado distribuidor, será necessário disponibilizar um endpoint de cliente para que a plataforma envie cadastros de clientes e atualizações, em nosso material de apoio é disponibilizado os arquivos WSDL e XSD’s para que sejam importados em um servidor IIS, por exemplo, para torná-lo público.

Pedido (Protocolo Soap) Seguindo o mesmo conceito de cliente, porém para receber pedidos da plataforma.

Pagamento (Protocolo Soap) Para receber as confirmações de pagamentos dos pedidos, cujo meio de pagamento é gerenciado pela infracommerce (Boleto antecipado, PIX, Boleto a Vista), notificamos esses pagamentos com status tanto de aprovação, quanto reprovação.



Pré-requisitos dos endpoints acima:

  • Devem estar protegidos por criptografia Basica (Basic64).
  • Devem ser expostos em protocolo HTTPS com um certificado válido (Recomendamos Modelo Wildcard, por atender subdomínios, e uma vez que, o distribuidor irá trabalhar com nos ambientes de homologação e produção. não é aceito certificado auto assinado como A1 e A3).
  • O certificado deverá ser compatível com protocolo de segurança TLS 1.2 ou superior.

    • Observação: Somente após a disponibilização dos web services de cliente, pedido e pagamento, atendendo aos requisitos acima, serão disponibilizados os acessos aos web services da Infracommerce. Lembrando que o retorno das respostas deve estar de acordo com os retornos requerido no manual de integração, conforme os exemplos de retornos.


    Web services que devem ser disponibilizados pela Infracommerce:


  • Estoque (Protocolo Soap) - Será disponibilizado pela infracommerce o endpoint de estoque para que o distribuidor realize os envios, contendo os produtos e valores de estoque.
  • Política de preços (REST) – Duas API’s, uma que gera o token de autenticação no no protocolo (GET) e outra API para o envio via (POST) para que o seller envie as políticas de preços.
  • Status de cliente (SOAP) – Para que o distribuidor envie Status de aprovação e reprovação parar os clientes enviados via plataforma.
  • Limite de crédito (SOAP) – Disponibilizar limite de crédito para os clientes desejados, lembrando que para validar essa funcionalidade, o cliente precisa constar com status aprovado.
  • Tracking (SOAP) – enviar as trackings de pedidos, sendo elas, separação, nota fiscal, faturamento concluído, entrega em andamento, entrega concluída, Cancelamento, Analise de credito aprovada e reprovada.

    • Ambiente:

    E Vale ressaltar, que o processo de implantação terá início no ambiente de homologação da plataforma e deverá seguir a mesma premissa do lado do seller a fim de evitar qualquer risco, e apenas após a conclusão de todas as etapas de implantação, o distribuidor será migrado para o ambiente de produção tendo em vista a execução do plano de testes dado como finalizado.


    Protocolo:

    Os webs services utilizam o protocolo SOAP para comunicação e apenas a integração de Política de preços segue o padrão REST, esses, serão utilizados possibilitar a integração do caminho plataforma > distribuidor e vice versa.


    Limite de requisições:

    Ao fazer uma requisição para o Middleware, independente do serviço (estoque, preço, política de preço, tracking, limite de crédito ou aprovação de clientes), o distribuidor deve limitar a lista de objetos em no máximo 50. Exemplo: o Seller precisa enviar preço para 200 produtos, neste caso, deverá dividir os envios em 4 requisições contendo 50 produtos.

    Para preços: Caso o distribuidor tenha tido alteração de estoque em 30 SKUs, deve-se enviar apenas uma requisição com os 30 SKUs ou Caso o distribuidor tenha tido alteração de estoque em 130 SKUs, deve-se enviar uma requisição contendo 50 SKUs. Após o retorno dessa primeira requisição deve-se enviar mais uma requisição contendo 50 SKUs e, após o retorno da segunda requisição, deve-se enviar mais uma requisição com os 30 últimos SKUs


    Pré - requisitos de Hardware e rede


  • Conexão estável de internet para não haver falha nas integrações.
  • Endpoints estáveis e sem alterações de URL ou senha (deve ser fixo e não dinâmico).
  • ERP apto para emitir notas fiscais nos ambientes de homologação e produção e gerenciar o cancelamento das notas durante os testes de homologação.
  • Não utilizar recursos ou programas que possam comprometer o processamento das integrações ou disputar recursos de hardware ou rede.

    • Observação: Toda alteração feita no ambiente do distribuidor como: Endpoint, credenciais, alteração no certificado digital, deverá comunicar com pelo menos duas semanas de antecedência, para que possamos prevenir futuros impactos na plataforma de integração.



    Recomendações

  • Sistema Operacional: Windows 7 ou Superior.
  • Processador: I5 8 geração ou superior
  • Memoria: 8GB ou superior
  • Espaço em disco: Minimo de 60GB disponível
  • Conectividade: Internet estavel, IP fixo para os web services
  • Banco de dados ou APIS: Gerenciamento do ERP utilizado.

    • 02 - Integração de estoque

    No modelo abaixo segue exemplo de envio de estoque contendo duas formas, uma requisição única para um produto e também uma requisição contendo mais de um produto. As integrações de estoque são realizadas no protocolo Soap. As Urls e dados de autenticãções são disponibilizados pela Infracommerce ao Seller.


    Regras

  • O distribuidor deve ser capaz de definir de quanto em quanto tempo a integração de estoque deve ocorrer, para que sejam enviadas em períodos pré-determinados e não a todo momento que há alguma alteração de estoque em algum SKU.
  • Só enviar a integração de atualização de estoque para SKUs que tenham alteração em sua quantidade de 5% ou mais desde a última integração enviada. Importante que o Seller não envie carga FULL a todo momento para não onerar a fila de integração.
  • A quantidade enviada deve ser em numeral, sem casas decimais e, consequentemente, sem ponto ou virgula.
  • Para distribuidores que trabalham com venda de caixas ou múltiplos, mas controlam seus estoques em sistemas internos e centro de distribuição por unidades, é necessário que os estoques sejam enviados para cada SKU respeitando a lógica de unidade ou caixa vendida no site. Ou seja, neste caso, o estoque deve ser calculado para a versão da caixa.

    • Exemplo aplicado para estoque em caixa:


    Se SKU1234 (exemplo fictício) tem 1000 unidades em estoque, mas é vendido em caixa com 4 unidades:

    Se for vender SKU1234C4 precisa enviar 250 como quantidade para o SKU1234C4, pois 1000/4=250. Se enviasse 1000, permitiríamos que o cliente compre até 1000 caixas de 4, totalizando 4000 unidades.

    Atenção: Este desenvolvimento só é necessário para distribuidores que enviam integrações de SKU’s de caixas que não são trabalhadas fisicamente pelo distribuidor

    O distribuidor deve ser capaz de desabilitar a trava, caso a tenha, de envio de estoque via integração apenas quando o SKU respeitar a regra de possuir estoque mínimo suficiente para a venda de pelo menos 3 dias (D-3) de venda.


    Apoio para o desenvolvimento

  • O Web Service responsável pela integração de estoque é oStockServices.
  • A operação responsável pela integração de estoque é asetStock.
  • Esse WebService fica hospedado no Middleware, portanto, deve ser chamada pelo Distribuidor. O material da apoio para construção dos WSDL e XSD estão presentes no item 11 do manual.

    04 - Politica de preços (JSON)

    O Distribuidor que optar pelo desenvolvimento deste item ao invés do dois, poderá trabalhar com preços diferenciados por cliente, devido a possibilidade de enviar N políticas de preços, cada politica possui os produtos, tabelas de preços e grupo de clientes que deseja vincular. Uma vez que a plataforma permite trabalhar com várias políticas, é possível aplicar os preços diferenciados.
    As Politicas de preços trabalham no protocolo REST, onde a plataforma disponibiliza ao seller duas APIS sendo elas:
    - API GET para que o seller resgataro Token de autenticação;
    - API POST para realizar o envio da politica de preços, utilizando o token como autenticação.

    Detalhes sobre regras de negócios para políticas de preços


    A politica com ID "DEFAULT" que veremos nos exemplos abaixo, se aplicará para todos os clientes que não fazem parte de outra politica de preços que o seller irá enviar. Ou seja, quando um novo cliente realizar o cadastro na plataforma, automaticamente estará na politica Default, e praticará os preços que foram enviados para ela (os preços a prazo, só estarão disponíveis para clientes que possuirem status de aprovação e limite de crédito concedido). Tendo em vista esse cenário, não há a necessidade de incluir todos os CNPJs que farão parte da politica DEFAULT. A plataforma considera que todos os CNPJs restantes que não estão relacionados a nenhuma outra politica, ingressem na DEFAULT.

    Cenários onde serão devem ser enviados clientes para politica Default: apenas em casos onde o distribuidor pretender mover um cliente de uma politica cluster para default.

    O distribuidor pode enviar outras políticas além da politica default, atribuindo para cada política: produtos, tabelas de preços e clientes, para que assim, pratique preços diferenciados para os seus clientes conforme a necessidade do negócio.


    Regras: Um cliente só pode fazer parte de uma política de preços, sendo assim: caso o mesmo esteja em uma política diferente do default, o mesmo enxergará apenas os preços que foram enviados para ela.

    - As requisições para plataforma, não devem ultrapassar 50 produtos por requisição, para que o pacote de integração não onere a fila da plataforma, o mesmo vale para inclusão de inclusão de novos clientes para a mesma.

    - As políticas de preços trabalham em forma de atualização, ou seja: Caso o distribuidor envie a primeira requisição para política default contendo 50 produtos, e uma outra requisição contendo somente 1 produto, a política irá conter 51 produtos. Isso vale para inclusão de clientes na política: Caso houver a necessidade de incluir um ou um grupo de clientes em uma determinada política já existente, não há a necessidade de enviar as requisições de clientes e produtos, deve ser enviado somente os clientes para a política, com isso ganhamos performance.

    05 - Integração de cliente

    A integração de clientes nasce na plataforma, no ato que um cliente realiza o cadastro no site, que é integrado com o Sintegra onde a maior parte dos dados cadastrais são extraídos por ele.

    Para que o distribuidor receber esses dados, é preciso que o mesmo disponibilize um web service exposto na web, para que os envios sejam integrados em seu sistema (banco de dados e ERP).

    Alguns Requisítos:

    - Para que o distribuidor esteja apto para receber essas informações, é necessário que o mesmo disponibilize um web servisse exposto na web, para que essa integração ocorra ao distribuidor, o padrão de envio dos dados são XML, no protocolo SOAP, e para criar esse componente, disponibilizamos os arquivo WSDL e XSD neste manual, no item material de apoio.

    - O endpoint precisa estar protegido por criptografia Basic 64 por questão de segurança.

    - O endpoint precisa possuir um certificado digital instalado.

    - O distribuidor precisa retornar um status inicial para os novos clientes enviados, veremos o exemplo mais adiante.


    Exemplo de requisição enviada da plataforma para o distribuidor.


    Nome do método: notifyCustomerCreationRequest




    Exemplo de retorno esperado:


    O retorno de Status PEN deverá ser retornado pelo distribuidor como resposta para todo novo cliente enviado via plataforma. Um cliente com esse Status, já está apto para efetuar pedidos com as modalidades de pagamentos gerenciadas pela Infracommerce.


    Caso o cliente entre no site posteriormente para atualizar algum dado cadastral, como telefone, e-mail e etc, a informação enviada da plataforma para o distruibuidor será uma nova notificação de cadastro contendo toda a requisição. Neste caso, é sugerido que o seller retorne o status atual que o cliente se encontra. (PEN, APV ou RPV).


    Pontos de atenção: Durante a implantação, será enviado para o distribuidor envios massivos de clientes, atendidos pelo seu range de CEP, por este motivo, é importante que toda estrutura do seller, de banco, infraestrutura, seja capaz de receber esses dados e integrá-los. Na etapa de homologação é possível antecipar essa carga, a fim de evitar quaisquer eventuais problemas na etapa de produção.

    06 - Integração de Status de Cliente

    Conceito: Uma vez que o seller recebe os cadastros de clientes em sua base e já possui o status (PEN que foi retornado durante a integração do cliente), é possível que o seller envie outros dois status na integração de Status de cliente, que possibilitam enviar dois status, sendo eles APV(aprovação) e RPV(reprovação).

    A integração de status de cliente é de origem distribuidor para plataforma, e para esta ação, será disponibilizado pela Infracommerce, o endereço do endpoint e credenciais para essa integração ocorrer.


    Tipos de Status:

    APV (aprovado): Possibilita que o cliente esteja apto para efetuar pedidos com a modalidade de boleto faturado (boleto a prazo), porém é preciso que o mesmo possua limite de crédito (consultar a etapa de integração de limite de crédito).

    08 - Integração de Pedidos.

    A integração de pedidos nasce na plataforma e é enviado para o distribuidor através do web service que deve ser disponibilizado pelo distribuidor para que os dados enviados sejam integrados em sua base de dados. O distribuidor devera disponibilizar e implementar o WSDL OrderINtegration.wsdl, presente em nosso manual de integração. Através desse serviço o Middleware padroniza a estrutura do pedido que é enviada para todos os Distribuidores.

    Um pedido feito na plataforma pode conter SKUs de mais de um Distribuidor. Dentro da estrutura desse pedido haverá uma entrega (nó delivery) para cada Distribuidor, contendo apenas os produtos e os valores referentes ao Distribuidor em questão.

    Antes do pedido ser integrado com os Distribuidores ele passa pelo processo de quebra de entregas, onde é gerada uma estrutura completa de pedido (objeto Order) para cada uma das entregas que o pedido possuir, gerando assim, uma estrutura completa de pedido que será enviada para cada um dos Distribuidores.


    OBS: os pagamentos de produtos referentes ao mesmo Distribuidor devem ser pagos integralmente com um único meio de pagamento, ou seja, os produtos referentes a um Distribuidor devem ser pagos integralmente com boleto faturado OU cartão de crédito OU boleto à vista. A única exceção a essa regra é o vale troca. O meio de pagamento vale troca pode ser combinado com qualquer outro meio de pagamento gerenciado pela plataforma Infrashop, sendo assim, podem existir casos nos quais o pedido enviado para o Distribuidor possua dois pagamentos: vale troca e cartão de crédito/boleto à vista e PIX. Pedidos pagos com boleto faturado não poderão ser combinados com vale troca.


    Pedidos feitos na forma de pagamento boleto antecipado/cartão o Distribuidor não pode emitir data de vencimento do pagamento na nota fiscal, pois essas formas de pagamento são tratadas pela infracommerce.


    As promoções aplicadas (criados) na plataforma Infrashop podem dar descontos no valor dos SKUs e também podem dar bonificações (SKUs que não são cobrados do lojista). Quando um SKU recebe desconto o seu preço de listagem (listPrice) é diferente do preço de venda (salePrice). Quando um SKU é bonificado ele é enviado com o tipo FFR e o preço de venda R$ 0,00, indicando que o Distribuidor deve aplicar o tratamento necessário para faturar tal SKU como uma bonificação.

    Caso haja alguma falha no envio do pedido ele é enfileirado para futuras tentativas. Após cinco tentativas falhas de integração o pedido automaticamente cai em um alarme para que passe por análise manual.

    Conforme requisitado pela plataforma, a integração do primeiro pedido do cliente sempre deve ser aceita pelo distribuidor.

    Devido ao fato da plataforma Infrashop trabalhar com filas distintas de integração, é necessário que o Distribuidor esteja preparado para que, eventualmente, o primeiro pedido do cliente seja informado antes da Notificação de Criação do Cliente.

    É necessário que o Distribuidor receba a requisição e retorne uma resposta imediatamente para o Middleware. Somente após retornar a resposta para o Middleware o distribuidor deve realizar a integração do pedido com o seu ERP.


    OBS: O pedido realizado através da plataforma, é integrado no ERP do seller de forma unificada, caso o distribuidor tenha necessidade do split de pedido a partir de pastas de vendedores ou indústrias, esta configuração deve ser desenhada e desenvolvida internamente pelo distribuidor.

    09 - Integração de Pagamento

    A integração de pagamento, tem como objetivo, informar ao distribuidor se os pedidos integrados em seu ERP, houveram pagamento aprovado ou reprovado, para que assim o ciclo de vida dos pedidos evoluam para as trackings (status atual do pedido). A integração de pagamento se aplica somente para pedidos cujo o meio de pagamento é gerenciado pela Infracommerce, sendo eles: Boleto antecipado, Cartão de crédito, PIX, Vale Troca, Vale Compra Agora (exclusivo para integrações da loja Compra Agora). Uma vez que o distribuidor recebe a confirmação de pagamento dos pedidos mencionados com o status de aprovação (AP), o mesmo deve seguir com o fluxo de trackings para o pedido, para seguir o fluxo de vida.


    Pré requisitos:

  • Distribuidor deve disponibilizar um web service exporto na web e que contenham os objetos presente no arquivo PaymentServices.wsdl
  • O web service deve ser protegido por usuário e senha Basic64
  • Possuir certificado digital SSL instalado. Recomendamos o modelo Wildcard.
  • O retorno da requisição deve estar dentro dos padrões estabelecidos no exemplo de retorno de requisição
  • ERP integrar e identicar corretamente se o pedido houve o pagamento confirmado ou reprovado

    • Importante: Pedidos cujo gerenciamento do pagamento é feito por responsabilidade do seller (Pedidos realizados com a forma de pagamento em boleto faturado) não recebem da plataforma confirmações de pagamento, pois é responsabilidade do distribuidor em decidir seguir ou não com o pedido, esses pedidos uma vez que recebidos já devem ser tratados no fluxo de trackings.

    Para pedidos que recebem confirmações de pagamento: a confirmação de pagamento é enviada em até 8 dias corridos, importante que neste prazo, o seller mantenha a reserva de estoque, por este motivo, é importante que o seller não efetue o cancelamento neste período.

    10 - Integração de tracking

    Contexto

    Trackings consiste na continuidade do fluxo de vida de um pedido. O web service que recebe trackings será exposto pelo middleware e o distribuidor deve realizar a integração para nossa plataforma, informando em qual status o pedido se encontra.
    O Web Service responsável pela integração de tracking é oTrackingServices.
    A operação responsável pela integração de tracking é acaptureTracking.
    Esse WebService fica hospedado no Middleware, portanto, deve ser chamada pelo Distribuidor.

    Para pedidos cujo meio de pagamento é gerenciado pela infracommerce (Boleto à vista, cartão de crédito, PIX e Vale, o processo de trackings só deverá ser realizado, uma vez que o mesmo receber a confirmação de pagamento com status de pagamento aprovado. Uma vez que esse fluxo é realizado, o distribuidor deverá enviar o fluxo de trackings para continuidade do pedido:


    Tipos de trackings - para o fluxo de pedidos cujo pagamento é gerenciado pela infracommerce (Sequencia)


    WMS - OPCIONAL– Separação e embalagem. Esse status deverá ser enviado para plataforma uma vez que o pedido entrou nesse status (A integração de WMS é opcional).

    NFS - OBRIGATÓRIA – Nota fiscal emitida: Nessa tracking devem ser informados todos os SKUS que foram farados, bem como outras informações que mostraremos no exemplo de requisição. Trata-se de uma trackingobrigatória para os pedidos que seguirão o fluxo de vida.

    FAT - Finalização de faturamento OBRIGATÓRIA- Essa tracking indica o encerramento do faturamento, uma vez que ela é enviada, não será integrada mais nenhuma nota fiscal. Essa tracking, indicamos que o seller envie depois de alguns minutos após a NFS para evitar algum erro de integração (Pois um pedido com a tracking FAT precisa ter uma NFS). Assim como A NFS, e uma trackingobrigatória, pois sem ela, a platafoma reconhecerá o faturamento do pedido. É essencial que todo pedido que possuir NFS, logo na sequencia, possuir também a tracking de faturamento (FAT) a fim de evitar divergencias em repasses.

    ETR- Opcional Entrega em andamento – Indica que a entrega está em andamento, deve ser utilizada para somente casos em que o distribuidor possibilita enviar informações de rastreio da entrega.

    ENT- Obrigatória - Entrega concluída - Essa tracking indica que o pedido foi entregue.

    Faturamento para pedidos que existem itens bonificados

    Cenários de pedidos que possuem kit bonificado: Se no pedido existir algum SKU bonificado (skuType = FFR que enviamos no XML do pedido), o valor dos produtos bonificados ( skuType = FFR ) devem ser zerado no tracking de Nota Fiscal Emitida ( enviando unitPrice = 0 para esses produtos), as demais SKUS cobradas do tipo normal (SkuType = NOR) seguem o fluxo normal, enviando o preço unitário real.

    Se no pedido existirem SKUs do tipo bonificados e cobrados, o envio do tracking de Nota fiscal emitida (NFS) deve ser informado tanto os produtos cobrados quanto aos bonificados, levando em consideração que os itens bonificados devem ser enviados com o valor unitário = zero para esses produtos, vale ressaltar que os cobrados devem ser enviados o valor do item normalmente.

    As informações que definem se o produto é ou não bonificado, são enviadas no XML do pedido, nela, existe tags com o nome kitSkuId, essa tag quando enviada com valor "NOR" define que o produto é cobrado, e quando enviada com valor "FFR", indica que o produto é bonificado, para esses casos FFR, o seller deve enviar o valor zerado para esses produtos. O distribuidor pode optar por enviar uma NFS a parte para os itens bonificados caso preferir (pois um pedido pode ter N notas ficais), porém vale ressaltar, que uma vez que a tracking FAT é enviada, nenhuma nota fiscal pode ser enviada posteriormente, pois ela indica a finalização de faturamento.

    Pontos de atenção

    Cada NFS só pode ter UM pedido, mas um pedido pode ter N notas fiscais, seja por uma nota bonificada, uma nota de venda ou um faturamento dividido em algumas notas. Sendo assim, ao faturar um pedido com mais de uma nota fiscal, será necessário gerar uma chave para cada nota e enviar todas as notas e chaves que estão relacionadas a esse pedido.

    Envio de chave na nota fiscal, o campo invoiceEletronicKey, representa o número da chave da nota fiscal que precisa ser enviado. Para cada nota fiscal é emitido uma única chave.

    A Tracking de FAT notifica a finalização do faturamento e os trackings de NF’s, devem, portanto, ser enviados antes da tracking de FAT. Não é obrigatório o número da chave. Caso aconteça o envio, é permitido que seja enviado somente uma chave entre todas notas emitidas.

    A tracking de FAT indica que não será emitida mais nenhuma nota fiscal para o pedido. Portanto, as trackings de NFS enviados após uma tracking de FAT não serão contabilizados para os valores das apurações mensais de investimento, podendo causar divergências nos repasses.

    Requisição FAT - Finalização do faturamento:
    No exemplo de requisição, é enviado a tracking FAT para o pedido 1826201, que indica a finalização do faturamento do pedido.
